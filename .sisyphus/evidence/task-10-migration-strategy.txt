# Task 10 - Stable Migration & Compatibility Strategy

## Migration Path Overview

### Current State
- All 19 crates published independently
- Users may directly depend on any crate
- No clear guidance on which to use

### Target State
- `open-lark` as canonical package
- Expert-tier crates available for specialized use
- Support-only crates deprecated (soft deprecation)

## Compatibility Strategy

### Phase 1: Soft Deprecation (RC → v0.15.x)
**Actions**:
- Add documentation headers to support-only crates
- No code changes
- No warnings emitted
- Full backward compatibility

**Timeline**: Immediate

### Phase 2: Documentation Emphasis (v0.16.x)
**Actions**:
- Expert-tier READMEs updated with canonical package notice
- Support-only crates get stronger deprecation notices
- Migration guide published
- Still no breaking changes

**Timeline**: v0.16.0 release

### Phase 3: Optional Consolidation (v0.17.x)
**Actions**:
- Merge `openlark-protocol` into `openlark-core`
- Keep re-export for compatibility: `pub use openlark_core::protocol::*;`
- Evaluate other support-only merges
- Breaking changes with migration path

**Timeline**: v0.17.0 release (if approved)

### Phase 4: Final Consolidation (v1.0)
**Actions**:
- Support-only crates removed as independent packages
- Functionality absorbed into canonical/expert-tier
- Clear migration tooling

**Timeline**: v1.0 release

## User Impact Assessment

### New Users
**Impact**: Minimal
- Documentation clearly points to `open-lark`
- Installation is straightforward
- Feature flags guide usage

### Existing Users (open-lark facade)
**Impact**: None
- No changes to API
- No changes to features
- Fully backward compatible

### Existing Users (expert-tier direct)
**Impact**: Minimal
- Crates continue to be published
- No API changes
- Optional migration to canonical package

### Existing Users (support-only direct)
**Impact**: Low-Medium
- Crates remain available during deprecation period
- Clear migration path to canonical package
- At least 6 months notice before removal

## Migration Guide Template

### From Support-Only to Canonical

**Before**:
```toml
[dependencies]
openlark-analytics = "0.15"
```

**After**:
```toml
[dependencies]
open-lark = { version = "0.15", features = ["analytics"] }
```

**Code changes**:
```rust
// Before
use openlark_analytics::search::SearchRequest;

// After
use open_lark::analytics::search::SearchRequest;
// or
use open_lark::prelude::*;
```

## Risk Mitigation

### Risk: Breaking existing users
**Mitigation**:
- Soft deprecation first (documentation only)
- Minimum 6-month notice period
- Major version bump for breaking changes
- Clear migration guides with code examples

### Risk: Feature bloat in canonical package
**Mitigation**:
- All features are optional
- Default features are minimal
- Users only compile what they enable
- No runtime overhead for unused features

### Risk: Confusion about which crate to use
**Mitigation**:
- Clear documentation hierarchy
- Single "Getting Started" guide
- FAQ addressing "Which crate should I use?"
- Sub-crate READMEs redirect to canonical

# Task 11 - Stable crates.io Publishing Strategy

## Current Publishing Approach

### Problems Identified
1. **Path dependencies without versions**
   ```toml
   openlark-core = { path = "crates/openlark-core" }  # Missing version!
   ```
   - crates.io requires version specifications
   
2. **Complex dependency graph**
   - 19 crates with interdependencies
   - Must publish in correct order
   - Manual process error-prone

3. **Version synchronization**
   - All crates use `version.workspace = true`
   - Good for consistency
   - But requires coordinated publishing

## Recommended Solutions

### Option 1: Manual Layered Publishing (Current)
**Process**:
1. Publish foundation crates: `protocol`, `core`
2. Publish business crates (in any order, parallel possible)
3. Publish `client`
4. Publish `open-lark`

**Pros**:
- Full control
- No new tooling

**Cons**:
- Time-consuming
- Error-prone
- Easy to miss order

### Option 2: cargo-smart-release (Recommended)
**Tool**: `cargo install cargo-smart-release`

**Usage**:
```bash
# Analyze and publish in correct order
cargo smart-release --workspace

# Dry run first
cargo smart-release --workspace --dry-run
```

**Pros**:
- Automatic dependency ordering
- Dry-run capability
- Changelog generation
- Version bumping

**Cons**:
- New tool to learn
- Additional dependency

### Option 3: Custom Publish Script
**Approach**: Bash script with ordered publish commands

```bash
#!/bin/bash
set -e

# Layer 0: Foundation
cargo publish -p openlark-protocol
cargo publish -p openlark-core

# Layer 1: Business crates (can parallelize)
for crate in openlark-auth openlark-communication ...; do
    cargo publish -p $crate &
done
wait

# Layer 2: Client
cargo publish -p openlark-client

# Layer 3: Facade
cargo publish -p open-lark
```

**Pros**:
- Customizable
- No external dependencies
- Transparent

**Cons**:
- Must maintain script
- No dry-run
- No changelog integration

## Recommendation: Hybrid Approach

### For RC Releases
- **Method**: GitHub Release only (no crates.io)
- **Rationale**: Avoid publishing complexity for prereleases
- **Status**: ✅ Already implemented

### For Stable Releases
- **Primary**: `cargo-smart-release`
- **Backup**: Custom publish script
- **Process**:
  1. Run `cargo smart-release --workspace --dry-run`
  2. Review proposed changes
  3. Run `cargo smart-release --workspace --execute`
  4. Verify all crates published

## Required Changes for Stable Publishing

### 1. Add Versions to Workspace Dependencies

**Before**:
```toml
[workspace.dependencies]
openlark-core = { path = "crates/openlark-core" }
```

**After**:
```toml
[workspace.dependencies]
openlark-core = { path = "crates/openlark-core", version = "0.15.0" }
```

Or use:
```toml
openlark-core = { path = "crates/openlark-core", version.workspace = true }
```

### 2. Update Release Workflow

Add `cargo-smart-release` to release workflow:

```yaml
- name: Publish to crates.io
  run: |
    cargo install cargo-smart-release
    cargo smart-release --workspace --execute
  env:
    CARGO_REGISTRY_TOKEN: ${{ secrets.CRATES_IO_TOKEN }}
```

### 3. Version Synchronization

Ensure all crates have consistent version:
```bash
# Check all versions match
grep "^version" crates/*/Cargo.toml | sort | uniq -c
```

## Testing Publishing

### Dry Run
```bash
cargo publish --dry-run -p open-lark
```

### Local Registry Test
```bash
# Use local registry for testing
cargo publish --registry local-test -p open-lark
```

## Rollback Plan

If publishing fails mid-way:
1. **Don't panic**: crates.io doesn't allow re-publishing same version
2. **Increment version**: Bump patch version in Cargo.toml
3. **Fix issue**: Address whatever caused failure
4. **Retry**: Publish again with new version

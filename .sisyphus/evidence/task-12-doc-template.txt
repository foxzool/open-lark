# OpenLark

> 飞书开放平台 Rust SDK - 企业级高覆盖率 API 客户端

### Basic Usage

```rust
use open_lark::prelude::*;

#[tokio::main]
async fn main() -> Result<()> {
    let client = Client::builder()
        .app_id("your_app_id")
        .app_secret("your_app_secret")
        .build()?;
    
    // Use the client...
    
    Ok(())
}
```

## 📦 Features

Enable only the features you need:

```toml
# Core services (auth + communication + docs + workflow)
open-lark = { version = "0.15", features = ["core-services"] }

# Add HR support
open-lark = { version = "0.15", features = ["core-services", "hr"] }

# Full feature set
open-lark = { version = "0.15", features = ["full"] }
```

## 🏗️ Architecture

OpenLark is organized as a unified SDK with optional features:

- **Core**: Essential infrastructure
- **Auth**: Authentication & token management
- **Communication**: IM, contacts, groups
- **Docs**: Documents, sheets, wiki
- **HR**: Human resources suite
- *(See all features in [Cargo.toml](./Cargo.toml))*

Individual crates are available for advanced use cases,
but most users should use this unified `open-lark` crate.
```

## Migration from Mixed Messaging

### Before (Inconsistent)
```markdown
# openlark-docs

OpenLark 文档模块...

## Installation

cargo add openlark-docs
```

### After (Unified)
```markdown
# openlark-docs

> **Note**: Use `open-lark` with `features = ["docs"]` instead.

## Direct Usage (Advanced)

cargo add openlark-docs
```

## Terminology Standards

### Always Use
- "OpenLark SDK" (the product)
- "`open-lark` crate" (the canonical package)
- "features" (for optional functionality)
- "modules" (for sub-crate organization)

### Never Use
- "install the HR crate" → "enable the hr feature"
- "use openlark-docs" → "use open-lark with docs feature"
- "crate X" without context → always reference SDK first

# Task 5 - Canonical Package and Crate Tiering Rules

## Definition: Three-Tier Positioning System

### Tier 1: Canonical Package (唯一默认入口)
**Package**: `open-lark`

**Definition**: 
The canonical package is the sole recommended entry point for all users. It provides a unified facade over all functionality through feature flags.

**Public Messaging**:
- "Install OpenLark SDK: `cargo add open-lark`"
- "Enable features as needed: `--features core-services,hr,meeting`"
- All documentation examples must use `open-lark` imports

**Documentation Priority**: 
- Highest - appears in all primary documentation
- First mention in every guide/tutorial
- Only installation method shown in README quickstart

**Compatibility Guarantee**:
- Full semantic versioning compatibility
- Breaking changes follow semver with migration guides
- Always published to crates.io for stable releases

---

### Tier 2: Expert-Tier Crates (场景化专业模块)

**Definition**:
Crates with clear standalone value for specific use cases. Users may directly depend on these when:
- They need only one specific domain (e.g., only HR APIs)
- They want to minimize compile-time dependencies
- They're building specialized tools

**Expert-Tier List**:
| Crate | Use Case | When to Use Directly |
|-------|----------|---------------------|
| `openlark-workflow` | Task/approval/board management | Building workflow automation tools |
| `openlark-meeting` | Video conferencing | Meeting room management systems |
| `openlark-platform` | Admin/directory services | Enterprise admin dashboards |
| `openlark-application` | App management | ISV/developer tools |
| `openlark-helpdesk` | Ticket/support system | Customer service platforms |
| `openlark-mail` | Email groups | Mail integration services |
| `openlark-security` | Security/ACS | Access control systems |
| `openlark-ai` | AI assistants | AI-powered applications |
| `openlark-cardkit` | Card messages | Bot/notification systems |
| `openlark-webhook` | Webhook bots | Custom bot implementations |

**Public Messaging**:
- "For specialized use cases, you may use individual crates"
- "Most users should prefer `open-lark` with features"
- Documentation header: "This is an expert-tier crate. New projects should consider using `open-lark` instead."

**Documentation Priority**:
- Medium - documented but not emphasized
- Clear pointer to canonical package in each crate's README
- API docs generated but with canonical package banner

**Compatibility Guarantee**:
- Published to crates.io
- Semver compatibility maintained
- May have independent minor version bumps

---

### Tier 3: Support-Only Crates (内部支撑模块)

**Definition**:
Crates that exist primarily as implementation details or have minimal standalone value. They remain published for compatibility but are not promoted as first-class products.

**Support-Only List**:
| Crate | Role | Reasoning |
|-------|------|-----------|
| `openlark-protocol` | WebSocket protobuf types | 3 LOC wrapper, no independent value |
| `openlark-analytics` | Analytics/search APIs | Small API surface (~14 APIs), niche use |
| `openlark-user` | User preferences | Narrow scope, typically bundled with auth |
| `open-lark-docs` | Legacy directory | Not a workspace member, historical artifact |

**Public Messaging**:
- Not mentioned in primary documentation
- README states: "Support crate for `open-lark`. Direct use not recommended."
- No standalone installation examples

**Documentation Priority**:
- Low - minimal documentation beyond API docs
- Clear deprecation-style warning on crate README
- Hidden from feature lists in main documentation

**Compatibility Guarantee**:
- Published to crates.io for compatibility
- Version follows workspace
- May be absorbed into larger crates in future major versions

---

## Tiering Decision Framework

Use this framework to determine a crate's tier:

```
1. Does the crate have >5k LOC and >50 public APIs?
   → If yes, consider expert-tier
   
2. Does the crate represent a complete business domain?
   (e.g., HR, Docs, Communication)
   → If yes, consider expert-tier
   
3. Does the crate have <2k LOC or <20 APIs?
   → If yes, likely support-only
   
4. Is the crate primarily generated code or wrappers?
   → If yes, likely support-only
   
5. Would a user ever reasonably want ONLY this crate?
   → If yes, expert-tier
   → If no, support-only
```

---

## Implementation Rules

### For RC Release (Immediate)
1. All documentation emphasizes `open-lark` as canonical
2. Expert-tier crates get "advanced usage" documentation section
3. Support-only crates get compatibility warnings

### For Stable Release (Future)
1. Consider consolidating support-only crates:
   - `openlark-protocol` → merge into `openlark-core`
   - `openlark-analytics` → keep or merge into `openlark-platform`
   - `openlark-user` → keep or merge into `openlark-auth`
2. Expert-tier crates may gain independent marketing pages
3. Canonical package gains comprehensive feature guides

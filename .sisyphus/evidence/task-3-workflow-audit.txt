# Task 3 - Release Workflow RC/Stable Behavior Audit

## Workflow File Analysis

**File**: `.github/workflows/release.yml`

## Crates.io Release Job Condition

```yaml
crates-io-release:
  needs: [validate, github-release]
  runs-on: ubuntu-latest
  # 只有当版本号不包含 -rc 或 -beta 时才发布到 crates.io
  if: ${{ !contains(github.ref, '-rc') && !contains(github.ref, '-beta') && !contains(github.ref, '-alpha') }}
```

## Behavior Analysis

| Tag Pattern | crates.io Release | Behavior |
|-------------|-------------------|----------|
| `v0.15.0-rc.1` | ❌ SKIPPED | Condition excludes `-rc` |
| `v0.15.0-beta.1` | ❌ SKIPPED | Condition excludes `-beta` |
| `v0.15.0-alpha.1` | ❌ SKIPPED | Condition excludes `-alpha` |
| `v0.15.0` | ✅ ATTEMPTED | No prerelease suffix, passes condition |

## Current Status

✅ **RC Releases**: Correctly skip crates.io (avoiding publish failures)
✅ **Stable Releases**: Will attempt crates.io publish

## Gaps for Stable Release

1. **Path Dependencies Without Versions**
   - Root `Cargo.toml` uses `openlark-core = { path = "crates/openlark-core" }` without version
   - crates.io requires version specifications for all dependencies
   - Fix needed: Add `version.workspace = true` to all internal dependencies

2. **Multi-Crate Publishing Order**
   - Need to publish dependencies before dependents
   - Order: protocol → core → business crates → client → open-lark
   - Tool recommendation: `cargo-smart-release` or manual publish script

3. **Version Synchronization**
   - All crates currently use `version.workspace = true`
   - Need to ensure consistent versioning across workspace

## Recommendations

### For RC (Current)
- No changes needed - workflow correctly skips crates.io for prereleases

### For Stable (Future)
1. Add version requirements to all workspace dependencies:
   ```toml
   openlark-core = { path = "crates/openlark-core", version.workspace = true }
   ```
2. Use `cargo-smart-release` for automated dependency-order publishing
3. Test publish with `cargo publish --dry-run` before actual release

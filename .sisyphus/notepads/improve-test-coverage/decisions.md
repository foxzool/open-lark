# Auth 模块测试增强 - 决策记录

## 决策 1：测试组织方式
**选择**：在 `#[cfg(test)]` 模块中添加独立测试函数
**理由**：
- 与现有代码风格保持一致
- 每个测试独立，便于调试
- 清晰的测试命名便于理解测试目的

## 决策 2：测试用例选择
**选择**：优先覆盖边界条件和边缘场景
**理由**：
- 现有测试已覆盖主要功能路径
- 边界值容易出错（0秒、刚好过期）
- 空集合处理常被忽视

## 决策 3：注释风格
**选择**：使用中文注释标记测试分区
**理由**：
- 与项目现有中文注释风格一致
- 便于团队理解测试组织
- 使用 "新增测试：" 前缀区分新旧测试

## 决策 4：验证方法
**选择**：使用 assert! 和 assert_eq! 组合
**理由**：
- assert! 用于布尔条件（is_valid()）
- assert_eq! 用于精确值比较
- matches! 用于枚举变体检查

## 决策 5：时间处理
**选择**：使用 std::thread::sleep 处理时间边界
**理由**：
- 创建后立即检查的令牌可能需要极短延迟
- 5毫秒足够让 SystemTime::now() 前进
- 避免测试脆弱性（flaky tests）

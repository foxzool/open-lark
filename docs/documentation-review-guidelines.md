# 文档审查指南

## 📋 概述

本文档建立 open-lark 项目的文档审查机制，确保所有文档保持最新、准确和有用。

## 🎯 审查目标

1. **确保准确性**: 文档内容与项目实际状态一致
2. **保持时效性**: 及时更新过时信息
3. **提高质量**: 确保文档清晰、完整、易读
4. **避免冗余**: 清理重复和过时内容

## 📅 审查清单

### 1. 技术准确性审查

- [ ] API覆盖率数据是否与实际代码一致？
- [ ] 功能标志状态是否正确？
- [ ] 项目阶段描述是否符合实际进展？
- [ ] 代码示例是否可运行？
- [ ] 配置说明是否正确？

### 2. 内容时效性审查

- [ ] 文档最后更新时间是否合理？
- [ ] 是否包含"TODO"、"FIXME"、"DEPRECATED"等过时标记？
- [ ] 版本号信息是否正确？
- [ ] 项目统计数字是否最新？

### 3. 结构和格式审查

- [ ] 文档结构是否清晰？
- [ ] 目录层级是否合理？
- [ ] 标题层级是否正确？
- [ ] 链接是否有效？
- [ ] 代码格式是否正确？

### 4. 价值评估审查

- [ ] 文档是否为用户提供实际价值？
- [ ] 内容是否有重复？
- [ ] 是否可以合并或简化？
- [ ] 目标受众是否明确？

## 🔄 审查频率

### 定期审查
- **月度审查**: 核心文档（README.md, 架构文档）
- **季度审查**: 所有文档全面检查
- **版本发布前**: 重要更新审查

### 触发审查
- **重大功能更新后**
- **架构变更后**
- **发现内容过时报告后**

## 📝 审查流程

### 1. 准备阶段
- 确定审查范围和优先级
- 准备审查清单
- 通知相关文档负责人

### 2. 执行阶段
- 按清单逐项检查
- 记录发现的问题
- 评估问题严重程度

### 3. 跟进阶段
- 创建问题跟踪清单
- 分配修复责任人
- 设定修复期限

### 4. 验证阶段
- 验证修复完成情况
- 更新审查记录
- 归档审查结果

## 🏷️ 文档分类和优先级

### 🔴 高优先级（核心文档）
- `README.md` - 项目主文档
- `CLAUDE.md` - 开发指南
- 架构文档
- API文档

### 🟡 中优先级（重要文档）
- 实施报告
- 分析报告
- 配置指南
- 部署文档

### 🟢 低优先级（辅助文档）
- 工具文档
- 示例文档
- 历史文档
- 临时文档

## 🚀 自动化检查

建议建立以下自动化检查：

### 1. 文档一致性检查
```bash
# 检查API覆盖率一致性
./scripts/check-api-coverage.sh

# 检查功能标志状态
./scripts/check-feature-flags.sh

# 检查文档链接有效性
./scripts/check-doc-links.sh
```

### 2. CI/CD 集成
- 在PR流程中加入文档审查检查
- 自动检测文档过时标记
- 定期生成文档状态报告
